    Solid Object Oriented Design

    Single Responsibility

    - is the calculator itself. A calculator has many fucntions we can call a class that we could flush out with more detail. However for our calculator we created just four classes were created that has a single responsibility. It serves one task which is a single responsibility. Making the work robust that makes it function well.

    OCP

    - for the calculator project the open and close principle should be open for extension and closed for others to work on the class created. Should be open to re-utilization and replaced but only for base modification. In this case the calcultor has define roles unless I or someone else change the methods. If new things are added it better suits us than the single responsibility.

    Liskov Substitution Principle

    - adding, subtracting, multiplication, and division should be able to code very well. In this principle with the calcultor we can keep the class in the family and able to call for it later if we need to.

    Interface Segregation

    – just try to keep it simple. The calculator can do a lot of things but dont want to add things we wont use. But in our case we need one call function for one interface. Do not want to confuse anyone looking at my calculator.

    Dependence Inversion

    – look for better coding making sure the dependency won’t be convoluted. So were looking for abstraction making the calculator work as one unit. Making our work easier to code and read for other developers.
